# 배포 & 운영 가이드

**버전:** v0.1
**작성일:** 2026-04-27

---

## 0. 개요

이 문서는 본 프로젝트의 배포 및 운영 절차를 정의한다.

**구성:**
- 모바일 앱 (Expo + EAS) → App Store + Google Play
- 어드민 콘솔 (Next.js) → Vercel
- 백엔드 (Firebase) → Firestore + Functions + Storage

---

## 1. 환경 분리

### 1.1 3개 환경

| 환경 | 용도 | Firebase 프로젝트 | 도메인 |
|---|---|---|---|
| **Production** | 실 사용자 | `app-prod` | (앱스토어), admin.example.com |
| **Staging** | 베타/QA | `app-staging` | admin-staging.example.com |
| **Dev** | 개발 | `app-dev` | localhost |

각 환경은 **독립된 Firebase 프로젝트** 사용.

## 2. 모바일 앱 배포 (EAS)

### 2.1 EAS 셋업

```bash
# 1. EAS CLI 설치
npm install -g eas-cli

# 2. Expo 계정 로그인
eas login

# 3. 프로젝트 초기화
cd apps/mobile
eas init
```

### 2.2 eas.json

```json
{
  "cli": {
    "version": ">= 13.0.0"
  },
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal",
      "env": {
        "EXPO_PUBLIC_ENV": "development"
      }
    },
    "preview": {
      "distribution": "internal",
      "channel": "preview",
      "env": {
        "EXPO_PUBLIC_ENV": "staging"
      }
    },
    "production": {
      "channel": "production",
      "env": {
        "EXPO_PUBLIC_ENV": "production"
      },
      "ios": {
        "autoIncrement": "buildNumber"
      },
      "android": {
        "autoIncrement": "versionCode"
      }
    }
  },
  "submit": {
    "production": {
      "ios": {
        "appleId": "you@example.com",
        "ascAppId": "1234567890"
      },
      "android": {
        "serviceAccountKeyPath": "./google-service-account.json"
      }
    }
  }
}
```

### 2.3 빌드 명령어

```bash
# 개발 빌드 (시뮬레이터/에뮬레이터)
eas build --profile development --platform all

# Beta (TestFlight + 내부 테스트)
eas build --profile preview --platform all

# Production
eas build --profile production --platform all

# App Store / Google Play 자동 제출
eas submit --profile production --platform ios
eas submit --profile production --platform android
```

### 2.4 OTA 업데이트 (EAS Update)

```bash
# JS/CSS만 변경된 경우 (네이티브 코드 변경 X)
eas update --channel production --message "FR-013 타임라인 다이어리 추가"
```

**제약**: 네이티브 패키지 변경 시 새 빌드 필수.

```json
{
  "projects": {
```

```bash

# Storage Rules
firebase deploy --only storage --project production

# 모두
firebase deploy --project production
```

## 5. CI/CD (GitHub Actions)

### 5.1 워크플로우 구성

```
.github/workflows/
├── ci.yml                    # PR + push: lint, type-check, test
├── mobile-build.yml          # tag mobile-vX.Y.Z: EAS build
├── admin-deploy.yml          # main 브랜치: Vercel은 자동
├── functions-deploy.yml      # main 브랜치: Firebase Functions
└── firestore-rules-deploy.yml
```

### 5.2 ci.yml 예시

```yaml
name: CI

on:
  pull_request:
  push:
    branches: [main, develop]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with:
          version: 9
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint
      - run: pnpm type-check
      - run: pnpm test
```

### 5.3 mobile-build.yml 예시

```yaml
name: Mobile Build

on:
  push:
    tags:
      - 'mobile-v*'

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      - run: pnpm --filter mobile build
      - uses: expo/expo-github-action@v8
        with:
          eas-version: latest
          token: ${{ secrets.EAS_TOKEN }}
      - run: |
          cd apps/mobile
          eas build --profile production --platform all --non-interactive
```

## 6. 배포 절차

### 6.1 모바일 앱 정식 출시 절차

1. **버전 결정** (Semantic Versioning)
2. **테스트 완료** (Staging 환경)
3. **변경 로그 작성**
4. **태그 생성**
   ```bash
   git tag mobile-v1.0.0
   git push origin mobile-v1.0.0
   ```
5. **GitHub Actions가 EAS Build 트리거**
6. **빌드 완료 후**
   - iOS: TestFlight 검토 → App Store 제출
   - Android: Play Console 검토 → 출시
7. **출시 후 모니터링**

### 6.2 어드민 콘솔
- `main` 브랜치에 머지 → Vercel 자동 배포
- 1~2분 내 라이브

### 6.3 Firebase Functions
- `main` 브랜치 머지 + Functions 변경 감지 → GitHub Actions
- `firebase deploy --only functions --project production`

### 6.4 Firestore Rules
- 가장 신중하게 (사용자 영향)
- Staging에서 충분히 테스트
- Production 배포 후 즉시 모니터링

## 8. 백업 / 재해 복구

### 8.1 Firestore 자동 백업
- 매일 자동 백업 (Firestore export to Cloud Storage)
- 30일 보관

```bash
# 수동 백업
gcloud firestore export gs://app-prod-backups/$(date +%Y%m%d)
```

### 8.2 Cloud Storage 백업
- 사용자 첨부파일 / 음성 파일
- Storage class: Nearline (저렴한 백업용)

### 8.3 복구 시나리오
- 데이터 손실 → 가장 최근 백업 복원
- RTO (Recovery Time Objective): 4시간
- RPO (Recovery Point Objective): 24시간

- 인덱스 최적화
- 비정규화로 쿼리 단순화
- 페이지네이션 (1회 50건 이하)
