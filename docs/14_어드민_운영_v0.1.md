## 0. 개요

이 문서는 본 프로젝트의 일상적인 어드민 운영 절차와 표준 대응 매뉴얼을 정의한다.

**대상 독자:**
- 어드민 콘솔을 사용하는 운영자 (현재는 펀제주 본인)
- 향후 추가될 운영팀 / 고객지원팀

**기준 문서**: PRD v0.6 / 13_배포_운영 / 인증 & 보안

## 2. 사용자 지원

### 2.1 문의 응대 SLA

| 우선순위 | 응답 시간 | 해결 시간 |
|---|---|---|
| 긴급 (결제, 보안) | 4시간 | 24시간 |
| 일반 | 24시간 | 7일 |
| 저우선순위 (개선 제안) | 7일 | - |

### 2.2 일반 문의 응대 흐름

```
[사용자 문의 접수] (앱 내 문의 / 이메일 / 스토어 리뷰)
    ↓
[어드민 콘솔 → 사용자 검색]
    ↓
[사용자 상세 페이지에서 상황 파악]
    - 가입일, 마지막 활동
    - 현재 Tier
    - 최근 30일 사용 패턴
    - 결제 이력
    - Sentry 에러 (해당 사용자)
    ↓
[FAQ 매칭 → 표준 응답]
    또는
[심층 분석 필요 → 추가 조사]
    ↓
[응답 작성]
    - 친절하고 명확하게
    - 한국어 (영어 사용자에게는 영어)
    - 다음 단계 안내
    ↓
[기록]
    - admin_logs/ 자동 기록
    - 향후 비슷한 케이스 참고용
```

### 2.3 자주 묻는 질문 (FAQ)

#### 2.3.1 음성 입력이 안 돼요
- 마이크 권한 확인
- 디바이스 STT 작동 여부 (다른 앱에서 시도)
- 인터넷 연결 확인
- → 해결 안 되면 Whisper 백업 활성화 (어드민이 사용자별 토글)

#### 2.3.2 AI 분류가 잘못됐어요
- 정상: 사용자가 직접 수정 가능
- 자주 발생: 해당 패턴을 AI 프롬프트 개선에 반영
- → 어드민이 프롬프트 수정 검토

#### 2.3.3 한도 초과인데 더 사용하고 싶어요
- Pro 업그레이드 안내
- 또는 어드민 임시 Override (특별 케이스)

#### 2.3.4 결제했는데 Pro 안 됐어요
- 영수증 검증 실패 가능성
- `subscriptions/` 컬렉션 확인
- 검증 재시도 (`verifyApplePurchase` 수동 호출)
- → 안 되면 Apple/Google 측 문제, 환불 후 재구매 안내

#### 2.3.5 데이터를 모두 삭제하고 싶어요
- 설정 > 계정 삭제 안내
- 30일 grace period 설명
- 즉시 삭제 원하면 어드민이 처리

### 2.4 사용자 정지 (FR-A06, Phase 4)

**정지 사유:**
- 이용약관 위반
- 사기성 결제
- 시스템 악용 (스팸 입력, AI 호출 폭주 등)
- 다른 사용자 신고 (협업)

**정지 절차:**
1. 어드민 콘솔 → 사용자 검색
2. "정지" 버튼 → 모달
3. 기간 선택 (7일 / 30일 / 영구)
4. 사유 입력 (필수)
5. 사용자 알림 여부 선택
6. 확인 → `adminSuspendUser` Cloud Function 호출
7. `admin_logs/` 자동 기록
8. 사용자에게 이메일 발송 (선택 시)

**복구 절차:**
1. 사용자 상세 → "정지 해제"
2. 사유 입력
3. 확인

### 2.5 환불 처리

#### 2.5.1 자동 환불 (조건 충족 시)
- 구매 후 7일 이내 + AI 호출 < 5회
- 사용자가 앱 내에서 요청 → 자동 처리
- 어드민 알림만 (확인용)

#### 2.5.2 수동 환불 (어드민 검토)
- 자동 조건 미달
- 환불 큐에서 24시간 이내 검토

**검토 기준:**
- 사용 기간
- 사용량 (Task 생성 수, AI 호출 수)
- 환불 사유
- 사용자 이력 (반복 환불 요청자?)

**처리:**
- 승인 → Apple/Google API 환불 요청
- 거절 → 사용자에게 사유 안내 (정중하게)

## 3. 콘텐츠 관리 (Phase 4)

### 3.1 공지사항 (FR-A07)

**작성 시점:**
- 신규 기능 출시
- 시스템 점검 안내
- 정책 변경 (이용약관, 개인정보 처리방침)
- 마케팅 캠페인 (할인 등)

**작성 흐름:**
1. 어드민 콘솔 → 콘텐츠 → 공지사항 → 새 공지
2. 제목, 본문 (Markdown), 대상 Tier 선택
3. 시작/종료일 설정
4. 푸시 알림 여부 선택
5. 미리보기 → 게시

**스타일:**
- 제목: 짧고 명확 (15자 이내)
- 본문: 핵심 메시지 + 행동 유도
- 친근한 톤 (앱 페르소나 유지)

### 3.2 이메일 템플릿 (FR-A07)

**관리할 템플릿:**
- 환영 (가입 완료)
- 결제 영수증
- 구독 갱신 안내 (만료 7일 전)
- 결제 실패 안내
- 14일 무료 체험 만료 안내
- 비활성 사용자 재참여 (30일 미접속)
- 정책 변경 안내
- 계정 삭제 확인

**작성 흐름:**
1. 어드민 콘솔 → 콘텐츠 → 이메일 템플릿 → 템플릿 선택
2. 제목, HTML 본문 편집
3. 변수 사용 (`{{userName}}`, `{{tier}}` 등)
4. 미리보기 (실제 변수로 렌더)
5. 테스트 발송 (본인 이메일)
6. 활성화

### 3.3 AI 프롬프트 관리 (FR-A07)

**관리 대상**: 모든 시스템 프롬프트 (분류, Draft, 챗봇, 만다라트, 리포트 등)

#### 3.3.1 프롬프트 수정 흐름
1. 어드민 콘솔 → 콘텐츠 → AI 프롬프트
2. Tier(Flash-Lite/Flash) + Task 선택
3. 현재 활성 버전 확인
4. "새 버전 만들기" → 편집
5. 변경 사유 입력 (필수)
6. **테스트** (실제 입력 5~10개로 시뮬레이션)
7. 저장 (활성화 X)
8. A/B 테스트 시작:
   - 50% 사용자에게 새 버전 적용
   - 1~2주 모니터링 (정확도, 만족도)
9. 결과 좋으면 → 100% 활성화
10. 결과 나쁘면 → 이전 버전 롤백

#### 3.3.2 프롬프트 변경 주의사항
- **정확도 영향**: 출력 형식 변경 시 Zod 스키마도 함께 수정
- **비용 영향**: 프롬프트가 길어지면 토큰 증가 → 비용 증가
- **캐싱 무효화**: 프롬프트 변경 시 시스템 프롬프트 캐시 자동 무효화
- **사용자 영향**: 갑작스러운 톤 변화는 사용자 혼란 → 점진적 변경

## 4. Tier 및 만다라트 템플릿 관리

### 4.1 Tier 정의 변경 (FR-A04)

**변경 가능 항목:**
- 가격 (USD, KRW)
- 한도 (음성, 텍스트, Project, 협업, 저장 등)
- 기능 활성화 토글

**변경 시나리오:**
- **즉시 적용** (immediate): 모든 사용자에게 즉시
- **신규 사용자만** (newUsersOnly): 기존 사용자는 보호
- **갱신 시** (onRenewal): 다음 결제 갱신 시

#### 4.1.1 가격 인상 절차
1. 30일 전 사용자에게 공지 (이메일 + 인앱)
2. `applyMode: 'newUsersOnly'`로 신규 가입자부터 적용
3. 기존 사용자는 다음 갱신 시 (또는 영구 grandfather 정책)
4. 어드민 콘솔에서 즉시 변경 X (사용자 신뢰 보호)

#### 4.1.2 한도 변경 절차
- 한도 **상향**: 즉시 적용 (사용자 이익)
- 한도 **하향**: 30일 전 공지 + 갱신 시 적용

### 4.2 만다라트 템플릿 관리 (FR-A08, Phase 4)

**관리 작업:**
- 신규 템플릿 추가
- 인기 템플릿 모니터링 (`usageCount`)
- 부적절한 템플릿 비활성화
- 시즌별 특별 템플릿 (예: "2026 신년 결심")

**템플릿 추가 흐름:**
1. 어드민 콘솔 → 템플릿
2. "새 템플릿 만들기"
3. 카테고리 선택 (health / career / finance / relationship / 학습 / 기타)
4. 중심 목표 입력
5. 8개 Section 입력
6. (선택) Sub-section / Task까지 입력
7. 미리보기
8. 활성화 / 비활성화

## 5. 인시던트 대응

### 5.1 인시던트 분류

| Severity | 정의 | 응답 시간 | 책임자 |
|---|---|---|---|
| **S1 (Critical)** | 서비스 전체 다운, 데이터 유출 | 즉시 | 어드민 본인 |
| **S2 (High)** | 핵심 기능 장애, 결제 다발 실패 | 1시간 | 어드민 본인 |
| **S3 (Medium)** | 일부 기능 장애 | 4시간 | 어드민 본인 |
| **S4 (Low)** | 사소한 버그 | 24시간 | - |

### 5.2 일반 대응 절차

```
[1. 발견]
   - Slack 알림 / Sentry 에러 / 사용자 신고
   ↓
[2. 평가]
   - Severity 결정
   - 영향 범위 (사용자 수)
   - 원인 추정
   ↓
[3. 격리]
   - 추가 피해 방지
   - 필요 시 기능 비활성화 (어드민 토글)
   ↓
[4. 사용자 안내]
   - S1/S2: 즉시 인앱 공지 + 이메일
   - S3: 인앱 공지
   - 정직하고 명확하게
   ↓
[5. 복구]
   - 핫픽스 배포 (필요 시)
   - 영향받은 사용자 데이터 복구 (있으면)
   ↓
[6. 사후 분석]
   - 원인 분석
   - 재발 방지 조치
   - 문서 업데이트
```

### 5.3 시나리오별 매뉴얼

#### 5.3.1 시스템 전체 장애 (S1)

**증상:**
- 모든 사용자 앱 사용 불가
- Cloud Functions 응답 없음
- Firestore 접근 불가

**대응:**
1. Firebase Status Dashboard 확인 (구글 측 장애 여부)
2. GCP 측 장애 → 대기 + 사용자 안내
3. 자체 문제 → 직전 배포 롤백 (`firebase functions:rollback`)
4. 사용자 안내 (인앱 공지 + 이메일):
   ```
   [긴급] 일시적 서비스 장애가 발생했습니다.
   복구를 위해 노력 중이며, 곧 정상화될 예정입니다.
   기록된 데이터는 안전합니다.
   ```
5. 복구 후 사후 안내

#### 5.3.2 데이터 손실 / 유출 (S1)

**손실 대응:**
1. 즉시 백업에서 복구
2. 영향 범위 정확히 파악
3. 영향받은 사용자에게 직접 안내

**유출 대응 (PIPA 의무):**
1. 즉시 어드민 모든 키 회전
2. 영향 범위 파악 (어떤 데이터, 몇 명)
3. **24시간 이내** 사용자 안내 + 개인정보 보호위원회 신고
4. 외부 조사 (필요 시)
5. 보안 조치 강화

#### 5.3.3 결제 실패 다발 (S2)

**증상:**
- Apple/Google 결제 실패율 급증

**대응:**
1. RevenueCat 대시보드 확인 (제품 ID 정상?)
2. App Store Connect / Play Console 확인 (제품 활성?)
3. 자체 verify 함수 정상 작동 확인
4. 일부 사용자만 영향 → 해당 영수증 분석
5. 전체 영향 → 결제 일시 중단 (UI 비활성화)
6. 사용자 안내

#### 5.3.4 AI 비용 폭주 (S2)

**증상:**
- 일일 AI 비용이 평소의 3배 이상

**대응:**
1. 어드민 대시보드 → 사용자별 비용 정렬
2. 이상 사용자 식별 (악용 가능성)
3. 의심 사용자 → 임시 한도 0 처리 (Override)
4. AI 호출 패턴 분석 (Sentry, Cloud Logging)
5. 시스템 결함이면 → 임시 차단 + 핫픽스

#### 5.3.5 다국어 응답 부적절 (S3)

**증상:**
- AI가 부적절한 언어 / 톤으로 응답
- 사용자 신고

**대응:**
1. 해당 입력 + 응답 분석 (Sentry/로그)
2. 안전 필터 강화 검토
3. 시스템 프롬프트 수정 (필요 시)
4. 영향받은 사용자에게 사과 + 보상 (필요 시)

## 7. PIPA / GDPR 데이터 요청 처리

### 7.1 사용자 데이터 요청 (Right to Access)

**요청 흐름:**
1. 사용자가 데이터 요청 (앱 내 또는 이메일)
2. 본인 인증 (이메일 인증 + 추가 확인)
3. `createExport` Cloud Function 호출 (`type: 'all', format: 'json'`)
4. 24시간 이내 다운로드 링크 제공
5. 30일 후 링크 자동 만료

### 7.2 데이터 삭제 요청 (Right to Erasure)

**요청 흐름:**
1. 사용자가 삭제 요청
2. 본인 인증
3. `users/{uid}.deletedAt` 설정
4. 30일 grace period 안내
5. 30일 후 cascade 삭제 (모든 관련 데이터)
6. 삭제 완료 이메일 발송

**주의사항:**
- 결제 영수증은 법정 기간(5년) 익명화 보존
- 백업 데이터는 30일 후 자연 삭제

### 7.3 데이터 수정 요청 (Right to Rectification)
- 사용자가 직접 수정 가능 (앱 설정)
- 못 하는 부분은 어드민이 처리

### 7.4 처리 기록

모든 PIPA/GDPR 요청은 다음 정보를 기록:
- 요청 일시
- 요청자 식별
- 요청 종류
- 처리 결과
- 처리 일시
- 처리한 어드민

→ `admin_logs/` 컬렉션에 자동 저장.

### 7.5 응답 시한
- **PIPA**: 10일 이내 (한국)
- **GDPR**: 30일 이내 (EU)
- 본 프로젝트 정책: **7일 이내** (양쪽 모두 충족)

## 9. 운영 도구 (어드민 콘솔 사용 가이드)

### 9.1 어드민 콘솔 메뉴 구성

```
1. 대시보드          → 시스템 전체 현황
2. 사용자            → 사용자 검색 / 상세 / 한도 수정 / 정지
3. Tier             → Tier 정의 변경
4. 통계              → 상세 통계 (DAU/MAU/MRR 등)
5. 콘텐츠
   ├ 공지사항       → 사용자 공지 발송
   ├ 이메일 템플릿   → 이메일 본문 관리
   └ AI 프롬프트     → 시스템 프롬프트 관리 + A/B 테스트
6. 템플릿            → 만다라트 템플릿 관리
7. 환불 큐           → 환불 요청 검토
8. 감사 로그         → 어드민 액션 이력
9. 설정              → 본인 계정 (TOTP 등)
```

### 9.2 자주 사용하는 작업 단축
- 사용자 검색: 상단 검색바 (Cmd/Ctrl + K)
- 빠른 액션: 사용자 상세 → 우측 액션 패널
- 다크모드: 우상단 토글

## 10. 운영팀 확장 시 (향후)

### 10.1 역할 분리

| 역할 | 권한 | Custom Claim |
|---|---|---|
| **super_admin** | 전체 권한 (어드민 추가/제거 포함) | `role: 'super_admin'` |
| **admin** | 운영 전반 (어드민 관리 제외) | `role: 'admin'` |
| **support** | 사용자 지원만 (Tier/시스템 설정 X) | `role: 'support'` |

### 10.2 신규 어드민 등록 절차
1. super_admin이 `admin_users/` 생성
2. Custom Claim 부여
3. 신규 어드민에게 등록 안내 이메일
4. TOTP 등록 (`adminSetupTOTP`)
5. 1주일 시범 운영 (모니터링)

### 10.3 어드민 퇴사 시
1. 즉시 Custom Claim 제거
2. Firebase Auth 계정 비활성화
3. 모든 키 회전 (해당 어드민이 접근 가능했던 모든 키)
4. `admin_users/{uid}.isActive = false`
